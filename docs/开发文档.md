# IotEdgeGateway 开发文档

## 1. 项目概述

IotEdgeGateway 是一个基于 RK3568 边缘节点和 STM32F411 下位机的物联网智能控制系统。项目目标不是单独完成某个传感器或页面，而是打通从下位机采集、无线通信、边缘网关处理、Web/Qt 可视化、规则联动、历史存储、视频监控到最终部署的完整链路。

整体数据流如下：

```text
STM32 采集传感器
  -> 串口 JSON
  -> ESP8266 WiFi/MQTT 模块
  -> RK3568 本地 MQTT Broker
  -> RK3568 网关主程序
  -> 设备注册表 / SQLite / 规则引擎 / WebSocket
  -> Web 页面和 Qt 客户端展示

Web/Qt 控制命令
  -> HTTP POST /api/control
  -> RK3568 网关发布 MQTT cmd
  -> ESP8266 订阅并转发串口
  -> STM32 解析 JSON
  -> 控制 LED / 电机 / 蜂鸣器
```

主要技术点：

| 分类 | 技术 |
| --- | --- |
| 下位机 | STM32F411、USART、AHT30、光照传感器、红外检测、PWM、电机、蜂鸣器 |
| 通信 | UART JSON、ESP8266 WiFi、MQTT |
| 网关 | RK3568、Linux、C++14、CMake、Mongoose、rapidyaml |
| 服务 | HTTP REST、WebSocket、MQTT Client、轻量 MQTT Broker |
| 数据 | JSON、YAML、SQLite |
| 视频 | V4L2、FFmpeg、GStreamer、RKISP、OV5695 |
| 展示 | Web 前端、Qt 客户端 |
| 部署 | 部署包、Buildroot package、overlay、自启动脚本 |

## 3. 开发阶段

### 3.1 STM32 下位机开发

目标：

1. 完成温湿度、光照、红外等传感器采集。
2. 完成 LED、电机、蜂鸣器等执行器控制。
3. 提供稳定的串口上报数据。
4. 支持接收上位机控制 JSON 并执行。

上行数据格式：

```json
{
  "temp": 27.7,
  "humi": 60.1,
  "light": 64,
  "ir": 0
}
```

下行控制格式：

```json
{
  "type": "control",
  "payload": {
    "led_on": 1,
    "led_br": 80,
    "motor_on": 1,
    "motor_sp": 30,
    "motor_dir": 0,
    "buzzer": 0
  }
}
```

开发中遇到的问题：

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| 串口输出异常字符 | 调试输出和无效字节混入 | 删除无意义输出，只保留 JSON 行 |
| cJSON 生成上报数据不稳定 | STM32 堆空间有限，动态分配风险高 | 上报数据改为固定 buffer + `snprintf` |
| 下发命令不执行 | 链路长，可能卡在 MQTT、ESP8266 串口或 STM32 解析 | 按 Broker 日志、ESP8266 串口、STM32 外设逐层排查 |
| Keil 工程文件损坏 | 工程文件被误删或路径变动 | 参考正确工程恢复 `.uvprojx/.uvoptx` 和必要源码 |

最终处理原则：

1. STM32 上报数据字段固定，使用字符串拼接更稳。
2. STM32 控制解析只处理项目需要的固定字段。
3. 串口发送继续使用原有封装函数，避免重写底层串口。
4. MCU 端尽量减少动态内存和复杂协议逻辑。

### 3.2 通信模块开发

项目最终通信选择为 ESP8266 WiFi + MQTT，原因：

1. 项目要求支持 MQTT，WiFi/MQTT 更容易和 RK3568 网关集成。
2. 相比 Zigbee，ESP8266 开发成本更低，调试链路更直观。
3. RK3568 可运行本地 MQTT Broker，ESP8266 可直接连接。
4. Web/Qt、规则引擎和数据库都围绕 MQTT topic 处理，架构统一。

通信模块职责：

```text
STM32 UART RX -> 解析一行 JSON -> 发布 MQTT telemetry
MQTT cmd SUB -> 收到控制命令 -> UART TX 给 STM32
```

MQTT 主题：

```text
上行：
iotgw/dev/telemetry/temp
iotgw/dev/telemetry/humi
iotgw/dev/telemetry/light
iotgw/dev/telemetry/ir

下行：
iotgw/dev/cmd/control
iotgw/dev/cmd/led
iotgw/dev/cmd/motor
iotgw/dev/cmd/buzzer
```

遇到的问题：

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| ESP 连接 MQTT 失败 | Broker IP 配错，ESP 和 RK3568 不在同一网络 | RK3568 和 ESP8266 连接同一热点，ESP 指向 RK3568 wlan0 IP |
| `127.0.0.1` 理解混乱 | 每台设备的本机地址都不同 | ESP 连接 RK3568 实际 IP，网关连接本地 Broker 用 `127.0.0.1` |
| 数据太频繁 | STM32 上报周期过短，终端刷屏 | 降低上报频率，Broker 日志重定向到文件 |
| 命令下发偶发失败 | 前端、网关、Broker、ESP、STM32 多层链路 | 按 topic 和串口输出逐层确认 |

### 3.3 RK3568 网关基础框架

网关主程序使用 C++14 开发，核心类为 `GatewayCore`。

启动流程：

```text
解析命令行参数
  -> 加载 YAML 配置
  -> 初始化日志
  -> 初始化设备注册表
  -> 加载传感器/执行器配置
  -> 加载规则配置
  -> 初始化 SQLite
  -> 连接 MQTT Broker
  -> 启动 HTTP/WebSocket 服务
  -> 进入 Mongoose 事件循环
```

基础模块：

| 模块 | 说明 |
| --- | --- |
| `ConfigManager` | 加载 YAML，并将嵌套 key 展平为 `network.http_api.port` |
| `Logger` | 支持控制台和文件日志 |
| `MongooseServer` | 封装 HTTP、WebSocket、静态文件服务 |
| `RestApi` | 统一分发 `/api/*` 请求 |
| `MqttClient` | 连接 Broker，订阅和发布 MQTT |
| `DeviceRegistry` | 保存设备状态和 topic 映射 |

遇到的问题：

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| Mongoose 接口不匹配 | 使用的版本没有 `mg_match_uri` | 改用 `mg_match` |
| C/C++ 混合编译链接错误 | Mongoose 是 C，业务是 C++ | Mongoose 单独编译为 `mongoose_lib` |
| YAML 嵌套读取不方便 | 配置是多层结构 | 递归展开为点分隔 key |
| 开发板缺少 CMake | RK3568 rootfs 不适合直接编译 | 在虚拟机/主机编译，再部署到板端 |
| `GLIBCXX` 版本不匹配 | 编译环境和板端运行库不一致 | 使用匹配工具链或 Buildroot 统一构建 |

### 3.4 设备管理与 API

设备统一抽象为 `DeviceEntity`：

```text
id              设备 ID
kind            sensor / actuator
transport       mqtt
telemetry_topic 上报主题
command_topic   控制主题
status          在线状态和最近数据
```

设备配置文件：

```text
config/devices/sensors.yaml
config/devices/actuators.yaml
```

当前设备：

```text
传感器：temp、humi、light、ir
执行器：led、motor、buzzer
```

主要 API：

```text
GET  /api/devices
GET  /api/devices/{id}
GET  /api/status
POST /api/control
POST /api/actuators/{id}/set
```

设计说明：

1. API 不直接读传感器，而是读取设备注册表。
2. 设备注册表由 MQTT telemetry 更新。
3. `/api/status` 用于 Web/Qt 快速刷新状态。
4. `/api/control` 用于统一控制多个执行器。

### 3.5 MQTT Broker 与 MQTT Client

项目中实现了轻量 MQTT Broker：

```text
src/tools/mqtt_broker/main.c
```

监听端口：

```text
0.0.0.0:1884
```

保留自研轻量 Broker 的原因：

1. 项目只需要基础 MQTT publish/subscribe。
2. 可减少对 mosquitto 的运行依赖。
3. 更适合后续 Buildroot 集成为项目自带服务。
4. 日志和启动脚本可统一管理。

网关自身也是 MQTT Client：

```text
订阅：iotgw/dev/telemetry/#
发布：iotgw/dev/cmd/#
```

最终运行关系：

```text
iotgw_mqtt_broker 负责消息转发
iotgw_gateway 负责业务处理
ESP8266 负责 STM32 与 MQTT 之间的桥接
```

### 3.6 Web 前端与 WebSocket

Web 页面位于：

```text
www/index.html
```

主要功能：

Web 数据来源：


WebSocket 作用：

```text
MQTT telemetry 到达网关
  -> 更新设备注册表
  -> 广播到 WebSocket
  -> 页面实时刷新
```

### 3.7 规则引擎

规则配置文件：

```text
config/rules/automation-rules.yaml
config/rules/alarm-rules.yaml
```

已实现能力：

1. 启动时自动加载规则。
2. 支持自动化规则和告警规则。
3. 支持启用、禁用、重新加载。
4. 支持比较运算符 `>`、`>=`、`<`、`<=`、`==`、`!=`。
5. 规则触发后可发布 MQTT 控制命令。

规则 API：

```text
GET  /api/rules
POST /api/rules/{id}/enable
POST /api/rules/{id}/disable
POST /api/rules/reload
```

遇到的问题：

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| 风扇刚打开就关闭 | 自动规则覆盖手动控制 | 增加 `manual_override_ms`，手动优先 |
| 告警和控制不好观察 | 日志刷屏 | 日志写文件，用 grep 过滤 cmd/rule |

手动优先配置：

```yaml
control:
  manual_override_ms: 30000
```

### 3.8 SQLite 历史数据

SQLite 模块位于：

```text
src/core/storage/sqlite_storage.hpp
src/core/storage/sqlite_storage.cpp
```

数据库路径：

```text
data/iotgw.db
```

配置：

```yaml
storage:
  enabled: true
  db_path: "data/iotgw.db"
  retention_days: 7
```

保存逻辑：

```text
收到 MQTT telemetry
  -> 解析 device_id 和 value
  -> 写入 telemetry_history
  -> Web/Qt 可通过 /api/history 查询
```

历史查询接口：

```text
GET /api/history?device_id=temp&limit=20
```

设计说明：

1. 只保存传感器遥测数据。
2. 不保存控制命令。
3. 使用保留天数清理旧数据，避免数据库无限增长。
4. 使用预编译语句降低 SQL 注入风险。

### 3.9 视频模块

视频模块位于：

```text
src/services/web_services/api/camera_api.cpp
```

当前实现：

```text
/dev/videoX
  -> V4L2
  -> FFmpeg 优先
  -> GStreamer 兜底
  -> /tmp/rk_stream/live.jpg
  -> GET /stream/live.jpg
```

视频接口：

```text
GET /api/camera/probe
GET /api/camera/start_stream
GET /api/camera/stop_stream
GET /api/camera/snapshot
GET /api/camera/start_record
GET /api/camera/stop_record
GET /stream/live.jpg
```

摄像头排查链路：

```text
排线方向
  -> I2C sensor 识别
  -> MIPI CSI
  -> DPHY
  -> RKISP
  -> /dev/videoX
  -> FFmpeg/GStreamer
  -> Web 显示
```

遇到的问题：

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| 有 `/dev/video0` 但无法取流 | V4L2 节点存在不代表 sensor 激活 | 继续查 I2C、DPHY、RKISP |
| `Unexpected sensor id(000000)` | 排线方向或 sensor 未识别 | 调整排线并重启 |
| `No link between dphy and sensor` | MIPI 链路未建立 | 检查 sensor 驱动、设备树、排线 |
| 终端视频日志刷屏 | FFmpeg/GStreamer 输出过多 | 重定向到 `stream/ffmpeg.log` 和 `stream/gstreamer.log` |

确认识别成功的关键日志：

```text
ov5695 2-0036: Detected OV005695 sensor
```

### 3.10 Qt 客户端

Qt 客户端和 Web 端功能一致，数据来源也一致：

```text
Qt -> HTTP REST API -> RK3568 Gateway
Qt <- WebSocket/HTTP <- RK3568 Gateway
```

规划功能：

1. 查看设备状态。
2. 串口模块。
3. 数据可视化模块。
4. 日志模块。
5. 数据库模块。
6. 控制单片机外设。
7. 显示视频流、录制、拍照。
8. 告警模块。

设计原则：

1. Qt 不直接控制 STM32。
2. Qt 和 Web 使用相同 API，保证状态同步。
3. Qt 页面风格对齐 Web，使用卡片、标签页、控制面板布局。
4. 数据可视化基于 `/api/history`，不是单纯进度条。

两种方式的区别：

| 方式 | 用途 | 特点 |
| --- | --- | --- |
| 部署包 | 快速演示和调试 | 不重刷系统，复制后运行 |
| Buildroot/overlay | 最终系统移植 | 程序、依赖、启动脚本进入 rootfs |

Buildroot 中需要体现：

1. FFmpeg、GStreamer、SQLite、Qt、V4L2 等依赖进入 rootfs。
2. `iotgw_gateway` 和 `iotgw_mqtt_broker` 作为项目 package 安装。
3. overlay 放置启动脚本和运行目录。
4. 开机后自动启动网关服务。

## 4. 关键问题与解决方案

### 4.1 页面没有数据

排查顺序：

```text
ESP8266 是否发布 telemetry
  -> Broker 是否有 PUB 日志
  -> Gateway 是否订阅 telemetry/#
  -> DeviceRegistry 是否更新
  -> /api/status 是否返回数值
  -> Web 是否访问正确 IP/端口
```

常用命令：

```bash
wget -qO- http://127.0.0.1:8081/api/status
wget -qO- http://127.0.0.1:8081/api/devices/temp
```

### 4.2 命令下发没有反应

排查顺序：

```text
POST /api/control 是否成功
  -> Broker 是否出现 iotgw/dev/cmd/control
  -> ESP8266 是否订阅 iotgw/dev/cmd/#
  -> ESP8266 是否向 STM32 串口发送
  -> STM32 是否解析 JSON
  -> 外设是否动作
```

常用命令：

```bash
grep "iotgw/dev/cmd" /tmp/iotgw_broker.log | tail -20
```

### 4.3 MQTT 地址混乱

结论：

```text
Gateway 连接本地 Broker：127.0.0.1:1884
ESP8266 连接 RK3568 Broker：RK3568 实际 IP:1884
浏览器访问页面：RK3568 实际 IP:8081
```

`127.0.0.1` 永远表示当前设备自己，不能跨设备访问。

### 4.4 规则覆盖手动控制

现象：

```text
手动打开风扇
规则立即触发 cooling_off
风扇马上关闭
```

解决：

```yaml
control:
  manual_override_ms: 30000
```

手动控制后 30 秒内，规则不覆盖对应执行器。

### 4.5 日志刷屏

处理方式：

```bash
./build/iotgw_mqtt_broker 1884 >/tmp/iotgw_broker.log 2>&1 &
./build/iotgw_gateway >/tmp/iotgw_gateway.log 2>&1 &
```

只查看命令：

```bash
grep "iotgw/dev/cmd" /tmp/iotgw_broker.log | tail -20
```

只查看遥测：

```bash
grep "iotgw/dev/telemetry" /tmp/iotgw_broker.log | tail -20
```

### 4.6 Buildroot 同步源码失败

问题：

```text
rsync: readlink_stat(".ssh_tmp/...") failed: Permission denied
```

原因：

Buildroot package 不应该同步完整项目目录中的 `.ssh_tmp`、`.git`、构建产物和部署包。

处理：

准备干净源码目录，只保留：

```text
CMakeLists.txt
cmake/
src/
config/
www/
```

12. Qt 客户端基础页面。
13. 部署包脚本。
14. Buildroot/overlay 集成方案。

需要继续稳定或收尾：

1. 视频流在不同摄像头节点上的兼容性。
2. Buildroot 最终镜像完整构建和烧录验证。
3. Qt 客户端细节测试。
4. 演示流程和答辩材料整理。

## 7. 项目经验总结

1. MCU 端应尽量减少动态内存，固定字段的 JSON 用固定 buffer 更稳。
2. MQTT Broker 放在 RK3568 本地，更符合边缘网关独立运行场景。
3. 页面没有数据时，不应先改页面，应先确认 MQTT telemetry 是否进入设备注册表。
4. 命令下发问题必须按链路逐层排查，不能只看前端按钮。
5. 自动规则必须考虑手动控制优先级。
6. 有 `/dev/video0` 不等于摄像头可用，还要确认 sensor、I2C、MIPI、RKISP 链路。
7. NFS 适合开发，不适合最终部署；最终应使用部署包或 Buildroot/overlay。
8. Web 和 Qt 应共用同一套 HTTP/WebSocket API，避免重复实现硬件协议。
