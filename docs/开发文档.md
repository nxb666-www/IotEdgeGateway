# IoT Edge Gateway 开发文档

## 项目概述

嵌入式系统课程项目，构建物联网边缘网关系统。STM32 端负责传感器数据采集和执行器控制，通过 Zigbee/串口与 RK3568 边缘网关通信。

**技术栈**：C++14 + CMake + Ninja + Mongoose + rapidyaml + MQTT

**整体进度**：约 37%

### 1.2 CMake 构建系统

**原理**：CMake 是构建系统生成器，读取 `CMakeLists.txt` 生成 Ninja/Makefile，再编译代码。

**关键配置**：
```cmake
# C++14 标准
set(CMAKE_CXX_STANDARD 14)

# FetchContent 自动下载第三方库
FetchContent_Declare(mongoose GIT_REPOSITORY ... GIT_TAG 7.15)
FetchContent_Declare(rapidyaml GIT_REPOSITORY ... GIT_TAG v0.10.0)

# Mongoose 编译为 C 静态库（因为是 C 语言写的）
add_library(mongoose_lib STATIC ${mongoose_SOURCE_DIR}/mongoose.c)

# 业务代码编译为 C++ 静态库
add_library(iotgw_common STATIC
    src/core/common/logger/file_logger.cpp
    src/core/common/config/config_manager.cpp
    src/core/device/manager/device_registry.cpp
    src/services/web_services/websocket/websocket_server.cpp
    src/gateway/gateway_core.cpp
)

# 可执行文件
add_executable(iotgw_gateway src/gateway/main.cpp)
target_link_libraries(iotgw_gateway PRIVATE iotgw_common)
```

**疑问**：为什么要分两个静态库？
**解答**：mongoose 是 C 语言写的，业务代码是 C++，分开编译避免语言混编问题。`iotgw_common` 链接 `mongoose_lib`，最终可执行文件链接 `iotgw_common`。

### 1.3 日志系统

**原理**：日志系统采用 Sink 模式，Logger 负责过滤和格式化，Sink 负责输出到不同目标（终端、文件）。

**核心设计**：
```cpp
// 日志级别：低于设置级别的日志不输出
enum class Level { Trace, Debug, Info, Warn, Error, Fatal };

// Sink 接口：决定日志输出到哪里
class Sink {
public:
    virtual ~Sink() = default;
    virtual void Write(Level level, const std::string& msg) = 0;
};

// ConsoleSink：输出到终端
// FileSink：输出到文件
```

**疑问**：为什么 ConsoleSink 和 FileSink 的声明在 `logger.hpp`，实现在 `file_logger.cpp`？
**解答**：声明放在头文件让其他代码能用 `#include "logger.hpp"` 创建对象，实现放在 `.cpp` 文件避免重复编译。如果把实现也放头文件，每个 include 的地方都会编译一遍，导致链接错误。

**疑问**：`std::lock_guard<std::mutex>` 是干啥的？
**解答**：自动加锁解锁。创建时锁住 mutex，函数结束（包括异常退出）自动解锁，防止多线程同时操作数据。

### 1.4 YAML 配置管理器

**原理**：把 YAML 嵌套结构展平为点分隔的键值对存到 map 里。

```yaml
# YAML 文件
network:
  http_api:
    host: 0.0.0.0
    port: 8080
```

展平后：
```
"network.http_api.host" → "0.0.0.0"
"network.http_api.port" → "8080"
```

**核心实现**：
```cpp
// 递归遍历 YAML 节点，用 "." 连接嵌套 key
static void FlattenNode(ryml::NodeRef node, const string& prefix,
                        std::map<string, string>& out) {
    if (node.has_children()) {
        for (size_t i = 0; i < node.num_children(); i++) {
            auto child = node.child(i);
            string key = prefix.empty()
                ? string(child.key().str, child.key().len)
                : prefix + "." + string(child.key().str, child.key().len);
            FlattenNode(child, key, out);
        }
    } else if (node.has_val()) {
        out[prefix] = string(node.val().str, node.val().len);
    }
}
```

**疑问**：`ryml::parse()` 和 `ryml::parse_in_arena()` 有啥区别？
**解答**：`ryml::parse()` 不存在！这个版本的 rapidyaml 用 `ryml::parse_in_arena()` 解析只读字符串，`ryml::parse_in_place()` 解析可修改的字符串。`in_arena` 会复制一份数据，`in_place` 直接在原字符串上操作（更快但会修改原数据）。

**疑问**：`using std::string` 为啥不能放头文件？
**解答**：放在头文件会污染所有 include 这个头文件的代码的命名空间，可能导致命名冲突。只能放 `.cpp` 文件里。

### 1.5 Mongoose Web 服务器封装

**原理**：Mongoose 是事件驱动的网络库，所有 I/O 在单线程中以非阻塞回调方式处理。

**核心流程**：
```
Start() 启动监听
    ↓
主循环每 50ms 调 Poll()
    ↓
有事件触发 → 调 on_event() 回调
    ↓
on_event 根据事件类型分发：
  - HTTP 请求 → 调 http_handler_
  - WebSocket 消息 → 调 ws_handler_
```

**疑问**：`c->fn_data` 是啥？
**解答**：连接关联的用户数据。`Start()` 里传了 `this` 指针进去（`mg_http_listen(&mgr_, addr, on_event, this)`），存在 `c->fn_data` 里。回调函数里取出来就能访问 `MongooseServer` 对象的成员。

**疑问**：为啥 `on_event` 要声明为友元？
**解答**：`on_event` 是独立函数（不是类成员），默认不能访问 `MongooseServer` 的 private 成员（`http_handler_`、`ws_handler_`）。加 `friend` 声明后就能访问了。

**疑问**：`static` 和 `friend` 为啥冲突？
**解答**：`friend` 声明把函数声明为外部函数（extern），但 `.cpp` 里写了 `static`（只在当前文件可见），两者冲突。去掉 `static` 就行。

### 1.6 REST API

**原理**：根据 URL 返回不同的 JSON 数据。

```cpp
// 匹配 URL → 返回响应
if (mg_match(msg->uri, mg_str("/api/health"), NULL)) {
    mg_http_reply(c, 200, "Content-Type: application/json\r\n",
        "{\"status\":\"ok\"}\n");
}
```

**疑问**：`mg_match_uri` 和 `mg_match` 有啥区别？
**解答**：`mg_match_uri` 在这个版本的 mongoose（7.15）里不存在，只有 `mg_match`。函数签名：`bool mg_match(struct mg_str str, struct mg_str pattern, struct mg_str *caps)`。

**疑问**：`mg_match` 的第三个参数 `caps` 是干啥的？
**解答**：捕获组。`/api/devices/#` 里的 `#` 是通配符，匹配的部分存到 `caps[0]` 里。比如访问 `/api/devices/temp`，`caps[0]` 就是 `"temp"`。

### 1.7 事件循环主框架

**原理**：`GatewayCore::Run()` 是程序主函数，按顺序初始化各模块，然后进入事件循环。

```cpp
int GatewayCore::Run(int argc, char* argv[]) {
    // 1. 解析命令行参数
    // 2. 读配置（ConfigManager）
    // 3. 初始化日志（Logger + ConsoleSink）
    // 4. 初始化设备注册表（DeviceRegistry）
    // 5. 启动服务器（MongooseServer）
    // 6. 设置 HTTP 回调（RestApi::HandleRequest）
    // 7. 注册信号处理（Ctrl+C 优雅退出）
    // 8. 事件循环 while(running) { server.Poll(50); }
    // 9. 退出清理
}
```

**疑问**：`signal(SIGINT, signal_handler)` 是干啥的？
**解答**：注册信号处理。`SIGINT` 是用户按 Ctrl+C 时发的信号，`SIGTERM` 是 `kill` 命令发的信号。收到信号后执行 `signal_handler` 把 `running` 设为 `false`，事件循环退出，程序优雅关闭。

**疑问**：lambda 表达式 `[](){}` 是啥？
**解答**：匿名函数。`[](mg_connection* c, mg_http_message* msg) { RestApi::HandleRequest(c, msg); }` 定义了一个没有名字的函数，直接传给 `SetHttpHandler`。等价于单独写一个函数再传函数指针，但更简洁。

## 阶段二：设备管理 + MQTT（完成度 30%）

### 2.1 设备数据模型

**原理**：定义设备的统一数据结构，所有设备（传感器、执行器）都用同一个模型。

```cpp
// 设备状态
struct DeviceStatus {
    bool online = false;           // 是否在线
    int64_t last_seen_ms = 0;      // 最后通信时间
    std::string last_payload;      // 最后收到的数据
    std::string last_topic;        // 最后通信的 MQTT 主题
};

// 设备实体
struct DeviceEntity {
    std::string id;                // 设备ID，如 "temp"
    std::string kind;              // "sensor" 或 "actuator"
    std::string transport;         // "mqtt"、"zigbee"、"uart"
    std::string telemetry_topic;   // 遥测主题
    std::string command_topic;     // 命令主题
    DeviceStatus status;           // 设备状态
};
```

### 2.2 设备注册表

**原理**：管理所有设备的"通讯录"，内部维护三个 map 实现快速查找。

```
by_id_["temp"] = 设备实体                           // ID 查设备
telemetry_topic_to_id_["iotgw/dev/telemetry/temp"] = "temp"  // 遥测主题查ID
command_topic_to_id_["iotgw/dev/cmd/led"] = "led"            // 命令主题查ID
```

**核心函数**：

| 函数 | 作用 |
|------|------|
| `Register()` | 注册设备，建立主题→ID 映射 |
| `Has()` / `Get()` | 查询设备 |
| `List()` | 返回所有设备 |
| `UpdateFromTelemetryTopic()` | 通过主题更新设备状态 |
| `UpsertMqttDeviceFromTopic()` | 更新或自动创建设备 |
| `ToJsonList()` / `ToJsonOne()` | 转 JSON 给 API 返回 |

**疑问**：为啥每个函数都要加锁？
**解答**：多个线程可能同时访问设备数据。比如线程1 MQTT 消息到达要写数据，线程2 Web 请求要读数据。不加锁会数据混乱或崩溃。`std::lock_guard` 函数结束自动解锁。

**疑问**：`Get()` 里为啥不直接调 `Has()`？
**解答**：`Has()` 和 `Get()` 都加了锁，`std::mutex` 不允许同一线程锁两次（会死锁）。所以 `Get()` 里只能自己查 `by_id_.find()`。

**疑问**：`UpdateFromTelemetryTopic` 和 `UpsertMqttDeviceFromTopic` 有啥区别？
**解答**：`Update` 只更新已注册设备，`Upsert` 更新或自动创建。实际用 `Upsert` 就够了。

**疑问**：函数后面的 `const` 是啥意思？
**解答**：承诺不修改类的成员变量。`const` 对象也能调这个函数，防止手误改数据。

### 2.3 REST API 扩展

按文档要求实现了所有设备相关接口：

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/health` | 健康检查 |
| GET | `/api/version` | 版本查询 |
| GET | `/api/devices` | 设备列表 |
| GET | `/api/devices/:id` | 设备详情 |
| POST | `/api/actuators/:id/set` | 下发命令 |

**解答**：对，API 只是调用注册表的方法返回数据。设备注册在 `gateway_core.cpp` 里，或者 MQTT 消息到达时自动注册。

## 数据流总览

```
传感器数据上行：
STM32 采集 → 串口/Zigbee → RK3568 网关 → MQTT 发布 → Broker
                                                        ↓
                                              Web 页面 WebSocket 显示

控制命令下行：
Web 页面点击 → POST /api/control → 网关 → MQTT 发布 → Broker
                                                        ↓
                                              执行器收到命令
```

## 通信链路

```
STM32 ←→ RK3568：串口 UART 或 Zigbee
RK3568 ←→ MQTT Broker：MQTT 协议
RK3568 ←→ Web 浏览器：HTTP REST + WebSocket
```
