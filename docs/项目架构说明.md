# 项目架构说明

## 1. 项目定位

IotEdgeGateway 是基于 RK3568 边缘节点和 STM32 下位机的物联网智能控制系统。系统完成传感器数据采集、执行器控制、规则联动、历史数据存储、视频预览、Web/Qt 展示等功能。

项目整体分为四层：

```text
传感器/执行器层  ->  通信接入层  ->  RK3568 边缘网关层  ->  Web/Qt 展示控制层
```

其中 STM32 负责直接连接传感器和外设，通信模块负责将 STM32 数据接入 MQTT，RK3568 负责本地消息转发、设备状态维护、规则处理、数据库存储和 Web 服务。

## 2. 总体架构

```text
STM32F411
  ├── AHT30 温湿度
  ├── 光照传感器
  ├── 红外检测
  ├── LED
  ├── 电机/风扇
  └── 蜂鸣器
        │
        │ UART JSON
        ▼
ESP8266 WiFi/MQTT 通信模块
        │
        │ MQTT
        ▼
RK3568 边缘节点
  ├── iotgw_mqtt_broker     本地 MQTT Broker
  ├── iotgw_gateway         主网关程序
  ├── Mongoose              HTTP / WebSocket / MQTT 基础库
  ├── SQLite                历史数据存储
  ├── 规则引擎              自动控制与告警
  ├── V4L2 + FFmpeg/GStreamer 视频模块
  └── Web 静态页面
        │
        ├── HTTP REST API
        ├── WebSocket 实时推送
        └── Qt 客户端访问相同 API
```

## 3. 主要模块

| 模块 | 位置 | 职责 |
| --- | --- | --- |
| 主程序 | `src/gateway` | 启动配置、日志、设备注册、MQTT、HTTP、规则、SQLite、视频接口 |
| MQTT 客户端 | `src/core/device/protocol_adapters/mqtt_adapter` | 连接本地 Broker，订阅遥测主题，发布控制命令 |
| MQTT Broker | `src/tools/mqtt_broker` | 基于 Mongoose 的轻量 MQTT Broker，监听 `1884` |
| 设备注册表 | `src/core/device` | 保存设备 ID、类型、主题、在线状态、最新 payload |
| REST API | `src/services/web_services/api` | 提供设备查询、控制、规则、历史、摄像头接口 |
| WebSocket | `src/services/web_services/websocket` | 向浏览器实时广播 MQTT 数据 |
| 规则引擎 | `src/core/control` | 根据传感器阈值自动发布执行器命令 |
| SQLite 存储 | `src/core/storage` | 保存遥测历史数据，支持 `/api/history` 查询 |
| 视频模块 | `src/services/web_services/api/camera_api.cpp` | 调用 V4L2 + FFmpeg/GStreamer 生成实时 JPEG、拍照、录制 |
| Web 页面 | `www/index.html` | 实时状态显示、外设控制、历史查询、视频控制 |
| 配置文件 | `config` | 环境、设备、规则、存储、网络配置 |

## 4. 数据上行链路

传感器数据从 STM32 到 Web 页面的流程如下：

```text
STM32 读取传感器
  -> 生成 JSON: {"temp":27.7,"humi":60.1,"light":64,"ir":0}
  -> ESP8266 串口接收
  -> ESP8266 拆分并发布 MQTT
  -> iotgw_mqtt_broker 转发
  -> iotgw_gateway 订阅 iotgw/dev/telemetry/#
  -> 更新 DeviceRegistry
  -> 写入 SQLite
  -> 规则引擎判断
  -> WebSocket 广播
  -> Web/Qt 页面刷新
```

MQTT 遥测主题：

```text
iotgw/dev/telemetry/temp
iotgw/dev/telemetry/humi
iotgw/dev/telemetry/light
iotgw/dev/telemetry/ir
```

推荐遥测 payload：

```json
{
  "device_id": "temp",
  "type": "sensor",
  "source": "stm32",
  "driver": "aht30",
  "data": {
    "value": 27.7
  },
  "ts": 123456
}
```

## 5. 控制下行链路

Web 或 Qt 下发控制命令的流程如下：

```text
Web/Qt 点击按钮
  -> POST /api/control
  -> iotgw_gateway 封装控制 JSON
  -> MQTT 发布 iotgw/dev/cmd/control
  -> ESP8266 订阅命令主题
  -> ESP8266 通过 UART 转发给 STM32
  -> STM32 解析 JSON
  -> 控制 LED / 电机 / 蜂鸣器
```

控制 payload：

```json
{
  "type": "control",
  "payload": {
    "led_on": 1,
    "led_br": 80,
    "motor_on": 1,
    "motor_sp": 30,
    "motor_dir": 0,
    "buzzer": 0
  }
}
```

为避免规则引擎立即覆盖人工操作，项目中配置了手动优先时间：

```yaml
control:
  manual_override_ms: 30000
```

人工控制后的 30 秒内，自动规则不会覆盖对应执行器。

## 6. Web 与 Qt 交互方式

Web 和 Qt 客户端都不直接访问 STM32，也不直接访问传感器硬件，而是统一访问 RK3568 上的网关服务。

```text
Web/Qt -> HTTP REST API -> GatewayCore -> DeviceRegistry / SQLite / MQTT
Web/Qt <- WebSocket <- GatewayCore <- MQTT telemetry
```

这样可以保证 Web 和 Qt 的数据来源一致，控制路径一致，避免两套客户端各自实现硬件协议。

## 7. 视频模块架构

摄像头属于 RK3568 本地外设，视频数据不经过 STM32。

```text
CSI/USB Camera
  -> /dev/videoX
  -> V4L2
  -> FFmpeg 优先 / GStreamer 兜底
  -> /tmp/rk_stream/live.jpg
  -> GET /stream/live.jpg
  -> Web/Qt 显示
```

拍照保存到：

```text
www/snap_YYYYMMDD_HHMMSS.jpg
```

录制保存到：

```text
www/rec_YYYYMMDD_HHMMSS.mp4
```

## 8. 部署架构

当前支持两种部署方式：

### 8.1 部署包方式

适合开发调试和演示：

```text
iotgw_package/
├── bin/
├── config/
├── www/
├── data/
└── start.sh
```

复制到开发板后执行 `sh start.sh start` 即可运行。

### 8.2 Buildroot/overlay 方式

适合最终系统交付：

```text
/opt/iotgw/bin/
/opt/iotgw/config/
/opt/iotgw/www/
/opt/iotgw/data/
/var/log/iotgw/
/etc/init.d/S99iotgw
```

Buildroot 负责把 GStreamer、FFmpeg、SQLite、Qt、V4L2 工具等运行依赖集成到 rootfs；overlay 负责放置启动脚本和运行目录。烧录后系统开机即可启动网关，不再依赖 NFS 或手动复制源码。

## 9. 关键设计说明

1. MQTT Broker 放在 RK3568 本地，方便边缘节点离线运行。
2. 网关程序使用 Mongoose 同时支撑 HTTP、WebSocket 和 MQTT client。
3. SQLite 只存遥测历史数据，不存控制命令。
4. Web/Qt 都通过同一套 REST API 和 WebSocket 访问网关。
5. 视频模块属于 RK3568 本地能力，和 STM32 控制链路分离。
6. 开发阶段可以用部署包，最终交付建议用 Buildroot/overlay 集成。
